# レッスン1: Spec駆動開発入門

Kiroの最大の特徴である「Spec駆動開発」を体験しましょう！このレッスンでは、ステアリングファイルでKiroの性格をカスタマイズし、自然言語から設計書を自動生成する流れを学びます。

## 🎯 学習目標

このレッスンを完了すると、以下のことができるようになります：

1. **ステアリングファイルの理解**: Kiroの応答をカスタマイズする方法
2. **Kiroの性格設定**: 語尾や話し方を自分好みに変更
3. **Spec駆動開発の理解**: Requirements、Design、Tasksの役割
4. **Requirements-first Workflow**: 自然言語から設計書を生成する流れ
5. **対話による要件の洗練**: Kiroとの対話で要件を明確化

## ⏱️ 所要時間

約30分

- このプロジェクトをKiroで開いていること

## パート1: 初期設定 - Kiroの性格をカスタマイズしよう

### ステアリングファイルとは？

ステアリングファイル（`.kiro/steering.md`）は、Kiroに対する永続的な指示を記述するファイルです。このファイルに書かれた内容は、Kiroがすべての応答で参照します。

**できること**:
- Kiroの性格や話し方を設定
- プロジェクト固有のルールを定義
- コーディングスタイルを指定
- 専門用語の説明を追加

### ステップ1: ステアリングファイルを開く

Kiroに以下のように話しかけてください：

```
.kiro/steering.mdを開いて、内容を見せてください。
```

ファイルが開かれ、以下のような内容が表示されます：

```markdown
# Kiroハンズオンチュートリアル - ステアリングファイル

## Kiroの性格設定

性格: フレンドリー
語尾: 〜です

（この部分を編集して、Kiroの応答が変わることを体験します）
```

### ステップ2: Kiroの性格を変更する

Kiroに以下のように指示して、性格と語尾を変更してみましょう：

**例1: カジュアルな友達風**
```
ステアリングファイルを編集して、Kiroの性格を「カジュアルで親しみやすい友達」に、
語尾を「〜だよ」「〜だね」に変更してください。
```

**例2: 丁寧な先生風**
```
ステアリングファイルを編集して、Kiroの性格を「丁寧で教育的な先生」に、
語尾を「〜ます」「〜ですね」に変更してください。
```

**例3: お嬢様風**
```
ステアリングファイルを編集して、Kiroの性格を「上品で優雅なお嬢様」に、
語尾を「〜ですわ」「〜ですの」に変更してください。
```

💡 **ヒント**: 自分で好きな性格を考えてもOKです！

### ステップ3: 変化を確認する

ステアリングファイルを編集したら、Kiroに話しかけて変化を確認しましょう：

```
こんにちは！今日の調子はどう？このプロジェクトについて教えて。
```

Kiroの応答が、設定した性格や語尾に変わっていることを確認してください。

**期待される応答の例**:

- **カジュアル風**: 「やあ！調子いいよ！このプロジェクトはKiroのハンズオンチュートリアルだね。」
- **丁寧な先生風**: 「こんにちは！調子は良好ですね。このプロジェクトについてご説明いたします。」
- **お嬢様風**: 「ごきげんよう！わたくし、とても元気ですわ。このプロジェクトについてお話しいたしますわね。」

### ✅ パート1の確認ポイント

- [ ] ステアリングファイルを開くことができた
- [ ] Kiroの性格と語尾を変更できた
- [ ] Kiroの応答が変化したことを確認できた
- [ ] ステアリングファイルの役割を理解できた

## パート2: Spec作成 - 自然言語から設計書を生成しよう

### Spec駆動開発とは？

Spec駆動開発は、Kiroの最大の差別化ポイントです。自然言語で「こんなものを作りたい」と伝えるだけで、Kiroが以下を自動生成します：

1. **Requirements（要件定義書）**: 何を作るのか、どんな機能が必要か
2. **Design（設計書）**: どう作るのか、アーキテクチャや実装方針
3. **Tasks（タスクリスト）**: 何から始めるのか、実装の手順

### Requirements-first Workflowの流れ

```
あなたのアイデア
    ↓
Kiroとの対話で要件を洗練
    ↓
Requirements.md（要件定義書）
    ↓
Design.md（設計書）
    ↓
Tasks.md（タスクリスト）
    ↓
実装開始！
```

### ステップ1: プロジェクトのアイデアを伝える

このチュートリアルでは、簡単なプロジェクトを作ってみましょう。以下のいずれかを選んでください：

**テーマ例1: シンプルなTodoアプリ**
```
シンプルなTodoアプリを作りたいです。
タスクの追加、完了、削除ができる機能が欲しいです。
Requirements-first Workflowで進めてください。
```

**テーマ例2: 名言ジェネレーター**
```
ランダムに名言を表示するアプリを作りたいです。
ボタンを押すと新しい名言が表示される機能が欲しいです。
Requirements-first Workflowで進めてください。
```

**テーマ例3: 簡単な計算機**
```
四則演算ができる計算機アプリを作りたいです。
足し算、引き算、掛け算、割り算ができる機能が欲しいです。
Requirements-first Workflowで進めてください。
```

💡 **ヒント**: 自分でアイデアを考えてもOKです！プログラミング初心者でも理解できるシンプルなものがおすすめです。

### ステップ2: Kiroとの対話で要件を洗練

Kiroがいくつか質問をしてきます。例えば：

- 「どんなユーザーを対象にしていますか？」
- 「どのプラットフォームで動かしますか？（Web、モバイル、デスクトップ）」
- 「データはどこに保存しますか？」

これらの質問に答えることで、要件が明確になります。わからない場合は「おすすめを教えて」と聞いてもOKです。

### ステップ3: 生成された設計書を確認

Kiroが以下のファイルを生成します：

- `.kiro/specs/[プロジェクト名]/requirements.md`
- `.kiro/specs/[プロジェクト名]/design.md`
- `.kiro/specs/[プロジェクト名]/tasks.md`

それぞれのファイルを開いて、内容を確認しましょう：

```
.kiro/specs/[プロジェクト名]/requirements.mdを開いて、内容を説明してください。
```

### ステップ4: 設計書の内容を理解する

**Requirements.md（要件定義書）**:
- ユーザーストーリー形式で「誰が」「何を」「なぜ」したいのかを記述
- EARS形式の受入基準で、機能の詳細を定義

**Design.md（設計書）**:
- システムの全体構成
- コンポーネントの役割
- データモデル
- 正確性プロパティ（テストで検証すべき項目）

**Tasks.md（タスクリスト）**:
- 実装の手順を段階的に記述
- 各タスクがどの要件に対応しているかを明記

### ✅ パート2の確認ポイント

- [ ] プロジェクトのアイデアをKiroに伝えることができた
- [ ] Kiroとの対話で要件を洗練できた
- [ ] Requirements.md、Design.md、Tasks.mdが生成された
- [ ] 各ファイルの役割を理解できた
- [ ] Spec駆動開発の流れを体験できた

## 💡 よくある質問

**Q: ステアリングファイルの変更が反映されない**
A: Kiroを再起動するか、プロジェクトを開き直してみてください。

**Q: Spec作成に時間がかかる**
A: 複雑なプロジェクトほど時間がかかります。シンプルなアイデアから始めることをおすすめします。

**Q: 生成された設計書が期待と違う**
A: Kiroに「requirements.mdの〇〇の部分を△△に変更してください」と指示して修正できます。

**Q: クレジットを節約したい**
A: 明確で具体的な指示を出すことで、再生成の回数を減らせます。
